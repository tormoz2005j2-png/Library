# Архитектура

## Цели

- отделить пользовательский интерфейс от хранения данных;
- не публиковать секреты вместе со статическим сайтом;
- сохранить возможность заменить сервер или СУБД;
- обеспечить безопасную миграцию старых локальных данных;
- оставить проект понятным без обязательного frontend-сборщика.

## Компоненты

### `apps/web`

Статическое одностраничное приложение без framework-зависимостей.

- `index.html` описывает доступную разметку и модальные окна;
- `assets/css/main.css` содержит все стили;
- `assets/js/app.js` отвечает за отображение, фильтрацию и формы;
- `assets/js/library-api-client.js` инкапсулирует HTTP, заголовки и ошибки;
- `assets/js/config.js` содержит только публичную конфигурацию окружения.

UI не знает, какая СУБД используется сервером. Его контракт — HTTP API и JSON.

GitHub Actions публикует в Pages только каталог `apps/web`. Серверный исходный код, миграции и конфигурация инфраструктуры не входят в статический артефакт.

### `services/library-api`

Серверная граница приложения. Она выполняет четыре задачи:

1. проверяет личный ключ доступа;
2. валидирует все входные данные;
3. преобразует клиентскую модель в модель хранения;
4. выполняет атомарные операции с базой.

Сейчас сервис реализован как Cloudflare Worker с D1. Провайдерские файлы локализованы внутри одного каталога, поэтому будущий Node.js/Fastify, Docker или другой backend можно добавить как новую реализацию того же контракта.

### `migrations`

SQL хранится отдельно от runtime-кода и версионируется. `0001_initial.sql` создаёт:

- `library_settings` — настройки и состояние первичной миграции;
- `library_items` — нормализованные поля библиотечной записи;
- индексы для основных сортировок и фильтров;
- триггер даты последнего изменения.

Денежные суммы хранятся целыми минимальными единицами (`*_cents`), чтобы исключить ошибки двоичной арифметики.

## Поток данных

```text
Пользователь
    ↓
apps/web (UI)
    ↓ JSON + X-Library-Token
library-api-client.js
    ↓ HTTPS
services/library-api
    ↓ проверка и преобразование
База данных
```

Добавление, изменение и удаление сначала подтверждаются сервером и только после успешного ответа применяются к состоянию интерфейса. Смена валюты выполняется оптимистично с откатом при ошибке.

## HTTP-контракт

| Метод | Путь | Назначение |
|---|---|---|
| `GET` | `/api/library` | Получить настройки и все записи |
| `PUT` | `/api/settings` | Изменить валюту |
| `PUT` | `/api/items` | Создать или изменить запись |
| `DELETE` | `/api/items/:id` | Удалить запись |
| `PUT` | `/api/library` | Атомарно заменить библиотеку из резервной копии |

Ошибки возвращаются как JSON `{ "error": "сообщение" }` с корректным HTTP-статусом.

## Безопасность

- серверный секрет отсутствует в Git и клиентских файлах;
- браузер хранит введённый ключ только в рамках вкладки;
- API сравнивает SHA-256 ключей и возвращает `401` без правильного ключа;
- CORS разрешает только настроенные origins, но не считается заменой авторизации;
- идентификаторы, статусы, типы, даты, оценки и суммы валидируются сервером;
- динамический текст экранируется перед вставкой в HTML;
- ответы с данными помечаются `Cache-Control: no-store`.

## Миграция и восстановление

При первом подключении к пустой БД клиент проверяет старый ключ `myLibrary.v6`. Если данные доступны в том же origin, они массово переносятся одной атомарной операцией. Если origin изменился, используется ручная пара «Экспорт» → «Импорт».

После инициализации пустая библиотека считается допустимым состоянием: удаление последней записи не запускает повторное заполнение демонстрационными данными.

## Замена текущего backend

Для отказа от Cloudflare новая реализация должна сохранить HTTP-контракт выше. После этого достаточно:

1. развернуть новый API;
2. настроить секрет на сервере;
3. изменить `apiUrl` в `apps/web/assets/js/config.js`;
4. перенести данные через JSON-экспорт и импорт.

Код `app.js`, HTML и CSS при этом менять не требуется.
